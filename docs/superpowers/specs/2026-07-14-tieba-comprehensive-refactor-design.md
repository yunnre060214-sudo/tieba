# 百度贴吧自动签到全面重构设计

## 1. 背景与目标

当前仓库可以完成单账号贴吧签到，但业务流程、网络请求、重试、通知和退出状态耦合在入口文件中。项目还存在 TBS 请求使用 HTTP、部分签到失败却可能让 GitHub Actions 显示成功、成功邮件语义不准确、两层重试互相叠加、没有离线自动化测试、提交编译产物、工作流版本老旧和文档不一致等问题。

本次重构不兼容旧配置，以个人单账号稳定使用为目标。新版每天北京时间 08:00 在 GitHub Actions 运行，也支持本地执行和诊断；每次运行都通过 QQ SMTP 发送结果邮件。只有所有贴吧均签到成功或已签到，并且邮件发送成功时，Workflow 才显示成功。

## 2. 范围

本次包含：

- 重写配置、贴吧请求、签到调度、结果报告、QQ 邮件和程序入口。
- 统一使用 HTTPS，并增加请求超时、错误分类和有上限的退避重试。
- 建立不依赖真实 BDUSS 和 QQ 邮箱的自动化测试。
- 更新 GitHub Actions、依赖、README、环境变量示例和项目元数据。
- 删除旧通知渠道、编译产物、JavaScript 备份、自动同步、自动保活和自动提交构建产物的工作流。

本次不包含：

- 多账号签到。
- Web 管理页面、数据库或历史统计。
- Server酱、Bark、Telegram、钉钉、企业微信和 PushPlus。
- 自动刷新 BDUSS、验证码识别或规避平台风控。

## 3. 用户可见行为

### 3.1 自动运行

- GitHub Actions 在每天 UTC 00:00，即北京时间 08:00，触发一次签到。
- 保留手动触发入口。
- 同一仓库同一时间只允许一个签到任务运行，后来的重复任务不与前一个任务并发。
- Workflow 最长运行 15 分钟，权限仅为读取仓库内容。

### 3.2 结果状态

一次运行只产生三种业务结果：

- `success`：所有贴吧均签到成功或已经签到。
- `partial_failure`：成功取得贴吧列表，但至少一个贴吧在最终重试后仍失败。
- `fatal_failure`：配置无效、BDUSS 失效、无法取得贴吧列表或 TBS，或出现其他无法继续的错误。

程序先形成最终报告，再尝试发送一次邮件。仅当业务结果为 `success` 且邮件发送成功时退出码为 0。部分失败、致命失败或邮件发送失败均使用非零退出码，使 GitHub Actions 的状态与真实结果一致。

### 3.3 邮件

- 每次运行都发送一封邮件，不区分成功或失败是否启用通知。
- 标题明确包含“成功”“部分失败”“执行失败”或“通知失败前的业务结果”。
- 正文包含北京时间、总数、成功数、已签到数、失败数、执行时长和失败原因。
- 部分失败时列出失败贴吧及最终错误；控制台日志默认对贴吧名称脱敏。
- 邮件发送失败时不得掩盖原始签到结果，控制台同时打印签到结果和邮件错误。

## 4. 配置

所有配置在程序启动时一次性读取并校验。配置无效时不访问贴吧签到接口。

必填变量：

- `BDUSS`：单个百度贴吧登录凭据。
- `MAIL_USERNAME`：QQ 邮箱账号。
- `MAIL_PASSWORD`：QQ 邮箱 SMTP 授权码。
- `MAIL_TO`：收件地址，允许一个地址。

可选变量及默认值：

- `BATCH_SIZE=5`：每批最多并发签到数，限制为 1 至 20。
- `BATCH_INTERVAL_MS=1500`：批次间隔，限制为 0 至 60000。
- `MAX_RETRIES=3`：单个贴吧瞬时失败后的最大额外尝试次数，限制为 0 至 5。
- `RETRY_BASE_DELAY_MS=3000`：第一次退避等待，限制为 500 至 60000。
- `REQUEST_TIMEOUT_MS=10000`：单次 HTTP 请求超时，限制为 1000 至 60000。

本地运行通过 `.env` 加载相同变量。示例文件只提供占位符，不包含可用凭据。

## 5. 模块设计

### 5.1 `config`

`loadConfig(env)` 返回一次校验完成且不可变的配置对象。调用方不再解析字符串、补默认值或判断范围。所有错误合并成一条可读的配置错误，避免修正一个变量后才发现下一个变量仍有问题。

### 5.2 `tieba-client`

该模块隐藏贴吧请求所需的 URL、请求头、Cookie、响应形状和错误转换，只提供验证登录、取得关注贴吧、取得 TBS 和签到单个贴吧的接口。

生产 adapter 使用 Axios。所有端点必须为 HTTPS，设置显式超时，不记录 BDUSS，不把包含 Cookie 的请求配置写入错误日志。HTTP 429、网络中断、超时和 5xx 被归为瞬时错误；认证失败、响应格式错误和明确的业务拒绝归为永久错误。

测试 adapter 在内存中按脚本返回成功、限流、超时或业务错误，以便测试完整签到流程而不访问百度。

### 5.3 `signin-runner`

这是核心深模块。其外部接口只接收已校验配置和贴吧 adapter，返回结构化运行报告。批次切分、并发、等待、重试、错误分类、计数和最终状态都隐藏在模块内部。

每个贴吧只保留一个最终结果。瞬时错误和“签到过快”可以重试；验证码、未加入贴吧、凭据失效及其他永久错误不重试。退避时间按指数增长并加入少量随机抖动，但不超过 30 秒。网络层不会再对同一个签到请求额外套一层独立重试，避免尝试次数失控。

登录验证、贴吧列表和 TBS 获取允许针对瞬时网络错误做有限重试；认证失败立即终止整次运行。

### 5.4 `report`

结构化报告是控制台、邮件和退出状态的共同事实来源。计数不再由多个可变变量分别维护，而是从最终贴吧结果计算，避免重试后成功数与失败数不一致。

该模块提供控制台文本和邮件文本两种渲染方式，不负责发送或输出。报告中保存原始贴吧名称；控制台渲染负责脱敏，邮件渲染显示失败贴吧的完整名称。

### 5.5 `email`

生产 adapter 使用 Nodemailer，经 `smtp.qq.com:465` 和 TLS 发送邮件。其接口只接收已经渲染好的标题和正文，不读取签到内部状态。

测试 adapter 只记录收到的邮件，用于断言成功、部分失败和致命失败都只发送一次且内容正确。

### 5.6 `main`

入口负责按顺序加载配置、创建生产 adapter、执行签到、渲染报告、发送邮件并设置退出码。入口捕获无法形成正常报告的异常，将其转换为 `fatal_failure` 报告后再尝试邮件通知。

入口不使用 `process.exit()` 强制中断，而设置 `process.exitCode`，让日志和资源清理有机会完成。

## 6. 数据流

1. 加载 `.env`（仅本地）并校验全部配置。
2. 验证 BDUSS。
3. 获取关注贴吧列表和 TBS。
4. 跳过已经签到的贴吧，其余按批次签到。
5. 只对可恢复错误执行有上限的退避重试。
6. 从每个贴吧的最终结果生成唯一报告。
7. 输出脱敏的控制台摘要。
8. 发送一次 QQ 邮件。
9. 根据业务结果和邮件结果设置退出码。

## 7. 错误处理规则

- 缺少或非法配置：不发起贴吧请求；如果邮箱配置本身有效，则发送致命失败邮件，否则只输出安全日志。
- BDUSS 失效：停止后续请求并发送致命失败邮件。
- 单吧永久失败：记录失败，不阻止其他贴吧继续签到。
- 单吧瞬时失败：按配置重试，最终失败后进入部分失败报告。
- 贴吧列表为空：视为成功，但邮件明确写明没有关注贴吧；不伪装成登录失效，因为登录已经单独验证。
- 邮件失败：不再次发送邮件，避免重复轰炸；记录安全错误并使 Workflow 失败。
- 未知异常：转换为不包含密钥、Cookie、完整 Axios 配置或堆栈敏感数据的致命失败报告。

## 8. 测试设计

测试使用 Vitest 和内存 adapter，覆盖外部可观察行为，不绑定内部实现。

最低测试场景：

- 配置默认值、边界值、缺失变量和非法数字。
- 全部签到成功、全部已签到和空贴吧列表。
- 首次失败后重试成功。
- 达到重试上限后部分失败。
- 永久业务错误不重试。
- BDUSS 失效后不再获取贴吧或签到。
- 报告计数在重试后仍准确，且每个贴吧只有一个最终结果。
- 成功、部分失败和致命失败邮件的标题与正文。
- 签到成功但邮件失败时退出状态仍失败。
- 控制台不出现 BDUSS、SMTP 授权码或完整贴吧名称。

本地和 CI 均运行：

- `npm run typecheck`
- `npm test`
- `npm run build`

真实 BDUSS 只用于用户手动触发或定时运行，不进入自动化测试。

## 9. GitHub Actions 与仓库整理

主工作流使用 `actions/checkout@v4`、`actions/setup-node@v4` 和 Node.js 22，执行 `npm ci`、类型检查、测试、构建和签到。依赖版本由 `package-lock.json` 锁定。

删除：

- `.github/workflows/build.yml`
- `.github/workflows/keep-alive.yml`
- `.github/workflows/sync-upstream.yml`
- `.github/last_activity.md`
- `dist/`
- `js-backup/`
- 旧的多渠道通知实现和类型

README 改为只描述新版需要的四个 Secret、手动运行、本地运行、结果状态和常见故障。`package.json` 的仓库地址、问题地址和主页改为当前 fork。

## 10. 完成标准

只有同时满足以下条件才算重构完成：

- 类型检查、自动化测试和构建全部通过。
- 测试证明成功、部分失败、致命失败和邮件失败的退出状态正确。
- 源码及工作流不存在 HTTP 贴吧端点、旧通知渠道或旧 Actions v3 引用。
- Git 历史之外不再跟踪 `dist/`、`js-backup/` 和自动保活文件。
- README 与实际 Secret 名称、运行时间和通知行为一致。
- 在不使用真实凭据的情况下完成全部自动验证，并给出一次手动 Workflow 验证说明。

## 11. 风险与取舍

百度贴吧使用的接口不是稳定公开接口，未来仍可能因响应格式或风控策略变化而失效。本次通过集中请求实现、响应校验和可测试 adapter 降低修复成本，但不承诺绕过验证码或平台限制。

QQ SMTP 是唯一通知渠道，因此 QQ 邮箱配置错误会使整个运行失败。这是有意设计：用户要求每次成功或失败都能收到邮件，静默丢失通知不应被标记为成功。
